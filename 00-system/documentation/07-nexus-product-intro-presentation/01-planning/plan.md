## Approach

**Dual-audience presentation strategy using 80/20 principle and stakeholder-specific messaging**

### Core Strategy:
Transform the 2400-line Beam-specific presentation into a focused 11-slide generic introduction by:
1. **Pareto Analysis**: Extract the 20% of content that delivers 80% of understanding
2. **Stakeholder Mapping**: Design slides that address BOTH audiences' specific needs simultaneously
3. **Live Demo**: Show (don't just tell) memory persistence with real example

### Workshop Structure:
**Format**: 1.5 hours (90 minutes total)

**Phase 1 - Foundation (30 min)**:
- 20 min: Presentation (11 slides - problems + solutions)
- 10 min: Workshop guide walkthrough + roadmap preview

**Phase 2 - Live Guided Onboarding (55 min)**:
- Everyone does Project 00 together (facilitator-led)
- Memory revealed naturally during Section 3 (goals.md creation)
- Real-time troubleshooting and questions

**Phase 3 - Wrap-Up (5 min)**:
- Recap what was built
- Next steps (Projects 01-03 async)
- Follow-up options

## Key Decisions

### Decision 1: Use Live Demo (Not Just Slides)
**Why**: 80% of belief comes from SEEING memory work
**How**: After 10 min slides, 5 min demo showing:
1. Session 1: Create goals.md with user's actual info
2. "done" ‚Üí close-session saves everything
3. Session 2 (new chat): Load goals.md ‚Üí AI remembers everything
**Impact**: Tangible proof > theoretical explanation

### Decision 2: Focus on 3 Core Solutions (Not 7+)
**Why**: Pareto - 3 concepts deliver 80% of value
**What**: Memory + Structure + Skills (team reusability)
**Cut**: Planning Mode, Instruction-Driven Architecture, Progressive Loading, Context Management, etc. (backend details)
**Impact**: Clear message, not overwhelming

### Decision 3: AI Agent Company Examples (Not Generic)
**Why**: Tailored examples resonate stronger than generic ones
**What**: Examples both teams relate to:
- Sales: Client proposal templates, pitch deck consistency
- Solutions: Client implementation workflows, analysis patterns, team knowledge sharing
- Universal: "Build once, reuse forever" (projects as templates)
**Impact**: Everyone sees themselves AND their team in examples

### Decision 4: Skills = Team Knowledge Sharing
**Why**: Biggest pain for Solutions is "built once, never retained, can't share"
**What**: Emphasize Skills as:
- Institutional knowledge preservation
- Team collaboration mechanism
- Easiest way to share best practices
**Impact**: Solutions team sees immediate ROI (billable hours saved)

### Decision 5: Template Reusability Pattern
**Why**: Users need to understand projects aren't one-time use
**What**: Explain "projects as sandboxes":
- Use template for different experiments
- Reuse structure for similar tasks
- Clone pattern for new clients
- Example: "Client A project ‚Üí Clone for Client B ‚Üí Same structure, different data"
**Impact**: Mindset shift from "one-off" to "reusable infrastructure"

### Decision 6: Transition to Live Onboarding (Not Q&A)
**Why**: Hands-on experience > questions
**What**: After slides, dive straight into Project 00 (everyone together)
**Impact**: Immediate action, higher conversion, learning by doing

### Decision 7: 11 Slides + Detailed Workshop Guide
**Why**: Slides stay focused (Pareto), Guide provides facilitator depth
**What**:
- Slides: 11 focused slides (20 min presentation)
- Guide: Timing, troubleshooting, FAQs, facilitation tips
**Impact**: Professional delivery, prepared for edge cases

## Socratic Questioning (Testing Assumptions)

**Q: Why 1.5 hours instead of shorter?**
A: User specified 1.5 hours. This allows full Project 00 completion (55 min guided) which demonstrates real value. Shorter = just theory, not hands-on.

**Q: Why guided Project 00 instead of letting them do it solo?**
A: First-time users need guidance. Real-time troubleshooting prevents frustration. Group learning creates team energy. Solo = higher drop-off rate.

**Q: What if technical issues derail the workshop?**
A: RISK: High probability with 10+ people setting up Claude Code live. MITIGATION: Pre-workshop setup session? Backup facilitator? Troubleshooting guide ready?

**Q: Why emphasize Skills so heavily for Solutions team?**
A: Their biggest pain is "work not retained, can't share." Skills directly solve this (build once, team shares). ROI is obvious (billable hours saved).

**Q: What if Sales team doesn't understand technical concepts?**
A: Visual/no-code approach mitigates this. Use business language ("templates" not "skills"), show folder structures not code. Workshop guide needs plain-language explanations.

**Q: Why 50/50 split assumption?**
A: User said "mixed." Assuming equal split until told otherwise. If actual split differs, slides still work (dual-audience design).

A: User explicitly said "templates yes those are good" and mentioned reusing for different experiments. This is core value prop, not assumption.

- Discover technical blockers AFTER commitment ‚Üí costly scope changes
